# Release process

**To cut a release: bump the version, commit, tag, push. CI does the rest.**

```bash
# 1. bump "version" in npm/package.json AND every optionalDependencies pin
# 2. update CHANGELOG.md
git commit -am "Release v0.2.0"
git tag v0.2.0
git push origin main --tags
```

Pushing the tag triggers `.github/workflows/publish.yml`, which runs the tests,
cross-compiles every binary, assembles the npm packages, publishes them, and
creates the GitHub release. You need no npm token and no OTP.

## How distribution works

Zeb is a Go binary distributed through npm. The binaries ship **inside** the npm
packages — nothing is downloaded at install time.

- `@zeb-link/zeb` — the package people install. Contains no binary: just
  `bin/zeb.js`, a shim that hands off to the platform package.
- `@zeb-link/zeb-<platform>-<arch>` — six packages, one native binary each,
  declaring `os` and `cpu` so npm installs only the one matching the host and
  skips the other five.

This is the layout esbuild and Biome use, and it is the same design `hitmaker`
uses. It is deliberate:

- **No install script.** Works under `--ignore-scripts`, which is a common
  default in CI and locked-down environments.
- **The binary is in the lockfile**, with an integrity hash. A postinstall
  downloader can't be verified this way.
- **No GitHub dependency at install time.** A GitHub outage or rate limit
  doesn't break `npm install`.
- **Works behind registry mirrors** (Artifactory, Verdaccio) that proxy npm but
  not raw GitHub.

`npm/packages/` is generated by `scripts/build-npm.sh` and is gitignored — it is
a build artifact, not source.

## Authentication: trusted publishing

CI publishes with **npm trusted publishing** (OIDC). There is no token: GitHub
Actions proves its identity to npm directly, and npm checks it against a trusted
publisher registered on each package.

This also means every published package carries a **provenance attestation** —
npm records which repo, commit, and workflow built the tarball, and shows a
"Built and signed on GitHub Actions" badge on the package page. That only works
because the build happens in CI from public source; a laptop publish can't
produce it.

### One-time setup, per package

Each of the seven packages needs its trusted publisher registered. A package
must **already exist** on npm before it can be configured, so a brand-new
package needs one manual publish first (see below).

⚠️ **Run `npm login` first.** `npm trust` requires real 2FA and **rejects** the
bypass-2FA token that a local publish uses. If `~/.npmrc` holds that token,
every call fails with `403 ... Two-factor authentication is required for this
operation` — even though `npm whoami` succeeds. `npm login` replaces it with a
2FA-verified session.

```bash
npm login    # REQUIRED — a bypass-2FA token cannot do this

for p in zeb zeb-darwin-arm64 zeb-darwin-x64 zeb-linux-arm64 zeb-linux-x64 zeb-win32-arm64 zeb-win32-x64; do
  npm trust github "@zeb-link/$p" \
    --file publish.yml --repo zeb-link/zeb --allow-publish --yes
done

npm trust list @zeb-link/zeb    # name it explicitly; a bare `list` reads ./package.json
```

Requires `npm@11.15.0`+.

`--file publish.yml` must match the workflow filename. Renaming
`.github/workflows/publish.yml` breaks publishing until every package's config
is updated.

## Manual publish (fallback)

Needed only to create a package that doesn't exist yet, or if Actions is down.

```bash
make release-check          # test, cross-compile, assemble, dry-run — publishes nothing
PUBLISH=1 make npm-publish  # for real
```

A local publish needs a granular token with "bypass 2FA" in `~/.npmrc`, or
`OTP=123456 PUBLISH=1 make npm-publish`. Note that npm is
[deprecating bypass-2FA tokens](https://gh.io/npm-gat-bypass2fa-deprecation)
for direct publishing — another reason to let CI do it.

A local publish produces **no provenance**. Prefer the tag.

## If a publish fails partway

npm versions are immutable, so a half-published release can't be repaired by
re-running the same version. If some platform packages published and the main
package didn't, bump to the next patch version and publish the whole set again.
The orphaned platform packages are harmless — nothing depends on them.

## Version rules

The version lives in `npm/package.json` and must match:

- every pin under `optionalDependencies` (the platform packages inherit it)
- the git tag (`v` + the version) — CI fails the release if they disagree

`make build` embeds it into `zeb version` via ldflags. Builds without ldflags
resolve the version from module build info instead:

| How it was built | `zeb version` reports |
| --- | --- |
| `make build` | the `npm/package.json` version, e.g. `0.1.0` |
| `go install …@v0.1.0` | `0.1.0`, from the git tag |
| `go install …@latest`, tagged | the newest tag |
| `go install …@latest`, untagged | a pseudo-version, e.g. `0.0.0-20260714185951-1cb3d7ca9d5b` |
| `go build ./cmd/zeb` in a clone | a pseudo-version, `+dirty` if the tree has uncommitted changes |
| built outside a repo, no VCS data | `dev` |

This is a second reason to tag: an untagged `go install …@latest` reports a
pseudo-version instead of a release number.

## Local verification

```bash
make build && bin/zeb version
make release-check
```
