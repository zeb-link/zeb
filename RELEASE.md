# Release process

Zeb is a Go binary distributed through npm. Unlike an installer that downloads
from a GitHub release, the binaries ship **inside** the npm packages — so
publishing works whether this repo is public or private, and going public later
changes nothing here.

The layout is the one esbuild and Biome use:

- `@zeb-link/zeb` — the package people install. Contains no binary: just
  `bin/zeb.js`, a shim that hands off to the platform package.
- `@zeb-link/zeb-<platform>-<arch>` — six packages, one native binary each,
  declaring `os` and `cpu` so npm installs only the one matching the host and
  skips the other five.

`npm/packages/` is generated by `scripts/build-npm.sh` and is gitignored — it is
a build artifact, not source.

## One-time setup

1. Log in to npm as a user who can publish to the `@zeb-link` scope:

   ```bash
   npm login
   npm whoami
   ```

2. Create the scope if it doesn't exist yet. On npmjs.com, create an
   organization named `zeb-link`. Scoped packages are private by default, which
   is why every publish below passes `--access public`.

## Release checklist

1. Pick the next semver version.

2. Update the version in `npm/package.json` — **both** the top-level `version`
   and every pin under `optionalDependencies`. They must match exactly; the
   platform packages inherit this version, and the main package pins them.

3. Update `CHANGELOG.md`.

4. Build and verify:

   ```bash
   make fmt
   make test
   make release-check
   ```

   `release-check` runs the tests, cross-compiles all six binaries, assembles the
   platform packages, and dry-runs every publish. Nothing leaves your machine.

5. Confirm `dist/` contains:

   ```text
   zeb-darwin-amd64
   zeb-darwin-arm64
   zeb-linux-amd64
   zeb-linux-arm64
   zeb-windows-amd64.exe
   zeb-windows-arm64.exe
   checksums.txt
   ```

6. Publish. Platform packages go first — the main package pins their exact
   versions, so it will not resolve until they exist on the registry:

   ```bash
   PUBLISH=1 make npm-publish
   ```

7. Commit and tag:

   ```bash
   git add .
   git commit -m "Release vX.Y.Z"
   git tag vX.Y.Z
   git push origin HEAD --tags
   ```

8. Smoke test from the registry:

   ```bash
   npm view @zeb-link/zeb version
   npm i -g @zeb-link/zeb
   zeb version
   zeb --help
   ```

## If a publish fails partway

npm versions are immutable, so a half-published release can't be repaired by
re-running the same version. If some platform packages published and the main
package didn't, bump to the next patch version and publish the whole set again.
The orphaned platform packages are harmless — nothing depends on them.

## Optional: GitHub release

Only useful once the repo is public, and only for people who want a binary
without npm. `dist/` already holds the assets and `checksums.txt`:

```bash
gh release create vX.Y.Z dist/* --title "vX.Y.Z" --notes-file CHANGELOG.md
```

Nothing in the npm install path depends on this.

## Local verification

`make build` embeds the npm package version into the binary via ldflags:

```bash
make build
bin/zeb version
```

Without ldflags, the binary resolves its version from the module metadata the Go
toolchain records at build time:

| How it was built | `zeb version` reports |
| --- | --- |
| `make build` | the `npm/package.json` version, e.g. `0.1.0` |
| `go install …@v0.1.0` | `0.1.0`, from the git tag |
| `go install …@latest`, tagged | the newest tag |
| `go install …@latest`, untagged | a pseudo-version, e.g. `0.0.0-20260714185951-1cb3d7ca9d5b` |
| `go build ./cmd/zeb` in a clone | a pseudo-version, `+dirty` if the tree has uncommitted changes |
| built outside a repo, no VCS data | `dev` |

This is why tagging matters: an untagged `@latest` install reports a
pseudo-version instead of a release number. Push the tag as part of the
release.
